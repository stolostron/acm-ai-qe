# Ubiquitous Language: ACM AI Quality Engineering

Shared vocabulary for all agents and skills in this repo. Read this before touching any app.

Bounded context: **ACM AI Quality Engineering** — a monorepo of Claude Code-driven QE tools for Red Hat Advanced Cluster Management. Combines deterministic Python (gather, validate, report) with Claude Code agents (investigate, classify, synthesize) connected to external systems via MCP servers.

**Repo design (where “why” lives):** There is no separate ADR directory — keep this file, [CLAUDE.md](CLAUDE.md), and [docs/](docs/) accurate when behavior changes. High-level choices today: **Stage 1** `gather.py` is deterministic Python (reproducible facts into `core-data.json`); **Stage 2** is Claude analysis; **Stage 3** `report.py` validates `analysis-results.json` against `apps/z-stream-analysis/src/schemas/analysis_results_schema.json` before emitting reports. **Knowledge** for ACM product facts is Markdown/YAML under `.claude/knowledge/` (optional Neo4j MCP for dependency graph), not an in-repo embedding store. **Portable skills** live under `.claude/skills/`; root **`.mcp.json`** is generated by `/onboard` and gitignored (machine-specific paths). **Session traces** are append-only JSONL under `.claude/traces/` (hooks). **Feature playbooks** use `base.yaml` plus version overlays merged by `id` in `FeatureKnowledgeService`. **Z-stream 12-layer** root-cause model and **acm-bug-hunter 10-dimension** audit model are intentionally different taxonomies — see [docs/acm-bug-hunter/IMPLEMENTATION-SPEC.md](docs/acm-bug-hunter/IMPLEMENTATION-SPEC.md) vs the analysis schema. For history beyond this blurb, use `git log` on the paths referenced in CLAUDE.md.

## System-Level Entities

| Entity | Definition |
|--------|-----------|
| **App** | One of 3 applications in `apps/`: z-stream-analysis, acm-hub-health, test-case-generator. Each has its own CLAUDE.md, agents, commands, and run artifacts. |
| **Portable Skill** | A tool-agnostic skill under `.claude/skills/` with a `SKILL.md` entry point. Works in Claude Code, Cursor, or any runtime that reads SKILL.md. 19 total. |
| **Subagent** | A Claude Code `Agent` subtask launched by a skill or orchestrator. Runs in isolation, writes artifacts to disk. Defined in `apps/*/.claude/agents/*.md`. |
| **Knowledge DB** | The unified markdown/YAML store at `.claude/knowledge/`. 14 ACM subsystems, 11 content categories (architecture, baselines, diagnostics, failures, etc.). This is the canonical cross-app knowledge source. App-local `knowledge/` dirs are operational subsets. |
| **MCP Server** | A Model Context Protocol server giving agents tools for external systems. Config in `.mcp.json` (gitignored, generated by `/onboard`). 8 servers: acm-source, jenkins, jira, polarion, neo4j-rhacm, acm-search, acm-kubectl, playwright. |
| **Run Directory** | Output directory for one pipeline execution. Z-stream: `apps/z-stream-analysis/runs/<timestamp>/`. TC-gen: `runs/test-case-generator/<JIRA-ID>/<timestamp>/`. |
| **Session Trace** | JSONL telemetry in `.claude/traces/` capturing tool calls, MCP interactions, subagent launches. For audit and debugging. |

## Z-Stream Analysis

Classifies Jenkins pipeline test failures by root cause.

### Classification (the primary output)

| Value | Meaning |
|-------|---------|
| `PRODUCT_BUG` | Defect in ACM product code |
| `AUTOMATION_BUG` | Defect in test automation code |
| `INFRASTRUCTURE` | Cluster/environment issue |
| `FLAKY` | Intermittent, no consistent root cause |
| `NO_BUG` | Expected or non-real failure |
| `MIXED` | Multiple root causes (requires `mixed_components[]`) |
| `UNKNOWN` | Insufficient evidence (confidence < 0.50) |
| `REQUIRES_INVESTIGATION` | Summary-level only. Never on individual tests. Means human review needed. Used in `overall_classification` field and report rendering, not in `per_test_analysis`. |

### Classification Path

| Path | Meaning |
|------|---------|
| `A` | Direct, high-confidence from primary evidence |
| `B1` | Via correlation with known patterns |
| `B2` | Via elimination/counterfactual reasoning |

### Root Cause Layer (12-layer model)

Maps where in the stack a failure originates. Used for infrastructure-focused classification.

| Layer | Name |
|-------|------|
| 1 | Compute/Storage |
| 2 | Networking/DNS |
| 3 | Certificates/TLS |
| 4 | etcd/API Server |
| 5 | Authentication/RBAC |
| 6 | Operators/Controllers |
| 7 | CRDs/Resources |
| 8 | Addons/Agents |
| 9 | Application Logic |
| 10 | Data/State |
| 11 | API/Integration |
| 12 | UI/Frontend |

### Other Z-Stream Entities

| Entity | Definition |
|--------|-----------|
| **Confidence** | Float 0.0-1.0. Below 0.50 forces UNKNOWN. Requires 2+ evidence sources. |
| **Evidence Tier** | Weight of evidence. Tier 1: direct observation (highest). Tier 2: correlation. Tier 3: inference. |
| **Per-Test Analysis** | One entry in `analysis-results.json` per failed test. Contains classification, path, confidence, root_cause_layer, evidence, JIRA correlation. |
| **Investigation Phases A-E** | Stage 2 substeps. A: ground and group. B: 12-layer root cause per group. C: cross-test correlation. D: validation and bias checks (PR-1 through PR-7, counterfactuals D-V5c/D-V5e). E: JIRA correlation. |
| **Environment Oracle** | Python service inside gather.py (Stage 0). Cross-references playbooks, KG, Polarion, cluster state to assess dependency health. Key types: `DependencyTarget` (operator\|addon\|crd\|component\|managed_clusters), `DependencyHealth` (healthy\|degraded\|missing\|unknown\|unchecked). |
| **Feature Area** | ACM functional domain for grouping tests. Examples: GRC, Search, CLC, ALC, Console, Observability, RBAC, Virtualization/Fleet-Virt, MTV, Submariner. |
| **Feature Playbook** | YAML defining per-feature: prerequisites, known failure paths, suggested classifications, component mappings. Base + version overlays (deep-merged by `id`). |
| **Diagnostic Trap** | Known pattern where investigation goes wrong. 14 traps documented. Used in hub health and z-stream. |
| **Core Data** | `core-data.json` — Stage 1 output. Jenkins metadata, test reports, console logs, stack traces, cluster context. Foundation for all subsequent stages. |
| **Cluster Diagnosis** | `cluster-diagnosis.json` — Stage 1.5 output. Health score, trap matches, operator/subsystem status, routing hints for Stage 2. |

### Z-Stream Pipeline Stages

```
Stage 0 (Oracle)    → Python inside gather.py  → dependency health map
Stage 1 (Gather)    → Python gather.py         → core-data.json, cluster.kubeconfig
Stage 1.5 (Diag)    → Claude agent             → cluster-diagnosis.json
Post-1 (Enrich)     → Claude data-collector    → enriched core-data.json
Stage 2 (Analysis)  → Claude agent (Phases A-E)→ analysis-results.json (schema-validated)
Stage 3 (Report)    → Python report.py         → Detailed-Analysis.md, HTML, SUMMARY.txt
```

Stage 3 uses NO MCP servers — purely deterministic.

## Test Case Generator

Produces Polarion-ready test cases from JIRA stories.

### TC-Gen Pipeline Phases

| Phase | Purpose | Artifact |
|-------|---------|----------|
| 0 | Input validation | — |
| 1 | JIRA investigation | `phase1-jira.json` |
| 2 | Code/PR analysis | `phase2-code.json` |
| 3 | UI discovery | `phase3-ui.json` |
| 4 | Context synthesis | `synthesized-context.md` |
| 5 | Live validation (optional) | `phase5-live-validation.md` |
| 6 | Test case writing | `test-case.md` |
| 7 | Quality review | `phase7-review.md`, `review-results.json` |
| 8 | Report generation | HTML + validation JSON |

### TC-Gen Entities

| Entity | Definition |
|--------|-----------|
| **Gather Output** | `gather-output.json` — Phase 1 Pydantic model with JIRA data, PR data, options. |
| **Analysis Result** | Phase 4 audit metadata: routes discovered, selectors found, Polarion coverage, complexity (low\|medium\|high), self_review_verdict (PASS\|FAIL). Not consumed by downstream phases — metadata for humans. |
| **Review Result** | Phase 7 output. Verdict: PASS\|FAIL. ValidationIssue severity: blocking\|warning\|suggestion. Category: metadata\|description\|setup\|steps\|teardown\|title. |
| **Console Area** | ACM console functional area for tagging and convention validation. Areas: governance, rbac, fleet-virt, cclm, mtv, clusters, search, applications, credentials, observability. |

|--------|-----------|

## Overloaded Terms

These terms mean different things depending on context. Always qualify them.

| Term | Ambiguity | Resolution |
|------|-----------|------------|
| **Stage vs Phase** | Z-stream uses "Stage" (0, 1, 1.5, 2, 3) for top-level pipeline and "Phase" (A-E) for investigation substeps. TC-gen uses "Phase" (0-8) for top-level pipeline. | Always qualify: "Z-stream Stage 2", "TC-gen Phase 7", "Investigation Phase D". Never bare "stage" or "phase". |
| **Classification** | Z-stream: test failure root cause (PRODUCT_BUG etc.). Bug hunter: implementation dimension verdict (CLEAN etc.). | Always specify which system. |
| **Knowledge** | Could mean: `.claude/knowledge/` (canonical DB), app-local `knowledge/`, feature playbooks, or Neo4j knowledge graph. | Use "Knowledge DB" for `.claude/knowledge/`, "playbook" for feature YAML, "KG" for Neo4j. |
| **Agent** | Could mean: Claude agent definition (`.claude/agents/*.md`), a subagent task, a portable skill, or the `AgentOrchestrator` Python class. | Use "Claude agent" for definitions, "subagent" for Agent subtasks, "skill" for `.claude/skills/`, "orchestrator" for the Python adapter. |
| **Pipeline** | Z-stream top-level (Stage 0-3), gather substeps, TC-gen (Phase 0-8), hub health (6-phase), or Jenkins CI pipeline. | Always qualify: "z-stream pipeline", "TC-gen pipeline", "health pipeline", "Jenkins pipeline". |
| **Oracle** | The Environment Oracle (Python service in gather.py) vs. general usage. | Always capitalize: "the Oracle" or "Environment Oracle". |
| **Area** | Feature area (GRC, Search — z-stream groupings) vs. console area (governance, rbac — TC-gen convention tags). | These overlap but are defined separately. Prefer the context-specific term. |
| **Report** | Stage 3 output, `Detailed-Analysis.md`, `analysis-report.html`, or `ClusterHealthReport` dataclass. | Use "z-stream report", "health report", or the specific artifact name. |
| **Validation** | Schema validation (JSON), environment validation (cluster state), review validation (TC-gen quality), artifact validation (`validate_artifact.py`). | Always qualify: "schema validation", "environment validation", "review validation", "artifact validation". |
| **Baseline** | YAML files in `baselines/` (expected healthy state) vs. general usage. | "Baseline YAML" or "healthy baseline" for the knowledge files. |

```
                        ─────────────────

                               ↓
              [cluster-diagnostic agent] → cluster-diagnosis.json

```

## Known Issues

- **Dead import**: `agent_orchestrator.py` line 28 imports `services.evidence_validation_engine` — no matching module exists. Dead code from a removed or never-created module.
- **Version drift in main.py**: `apps/z-stream-analysis/main.py` says v3.5 in its docstring and argparse description. Docs and CLAUDE.md say v4.0. The docs are authoritative — main.py labels are stale.
- **MCP rename (resolved, 2026-05)**: Legacy `mcp/acm-ui-mcp-server/` was removed from the repo; use `mcp/acm-source-mcp-server/` and the `acm-source` MCP server key (aligned with Cursor). `apps/acm-hub-health/setup.sh` and sample `.mcp.json` files under apps were updated accordingly.
- **Dual knowledge paths**: `.claude/knowledge/` is the canonical cross-app Knowledge DB (14 subsystems, 11 content categories). `apps/z-stream-analysis/knowledge/` is an operational subset with YAML files specific to z-stream runtime (components, baselines, prerequisites, etc.). Both are actively used but serve different roles.
