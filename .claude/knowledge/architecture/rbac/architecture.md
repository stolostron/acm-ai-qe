# RBAC -- Architecture

## What RBAC Does

Fine-grained RBAC extends standard Kubernetes RBAC with ACM-specific multicluster
role management. Enables administrators to assign scoped permissions across
managed clusters -- granting users access to specific clusters, cluster sets,
and projects without cluster-admin privileges.

The feature spans four layers: MCRA operator (hub), ClusterPermission CRD (hub),
ManifestWork propagation (hub-to-spoke), and Role/RoleBinding enforcement (spoke).

---

## Feature Flag

Fine-grained RBAC requires the `fine-grained-rbac` MCH component, which is
**disabled by default**. When disabled:
- User Management tab does not render in console
- RBAC API endpoints return 404
- MCRA operator is not deployed

When enabled:
- User Management tab appears under Access Management
- MCRA operator deployed to MCH namespace
- acm-roles addon deployed to managed clusters
- console-api serves RBAC endpoints

## Core CRDs

### MultiClusterRoleAssignment (MCRA)

- **API Group:** `rbac.open-cluster-management.io/v1alpha1`
- **Scope:** Cluster

Hub-side CR defining a role assignment. Specifies:
- **Subject:** User, Group, or ServiceAccount
- **Scope:** Global, ClusterSet(s), or specific Cluster(s)
- **Roles:** One or more ClusterRoles to assign
- **Projects:** Optional project-level scoping within clusters

MCRA is the source of truth for intent. The MCRA operator reconciles it into
ClusterPermission resources.

### ClusterPermission

- **API Group:** `rbac.open-cluster-management.io/v1alpha1`
- **Scope:** Namespaced

Hub-side CR in each target managed cluster's namespace. Generated by MCRA
operator from MCRA spec. Contains:
- `clusterRole` -- single role reference (original field)
- `clusterRoleBindings` -- array of roles (newer field for multi-role assignments)
- `roles` / `roleBindings` -- namespace-scoped role references
- Subject information

ClusterPermission is propagated to spokes via ManifestWork.

## Prerequisites

- `cluster-permission` enabled in MCH (enabled by default)
- `fine-grained-rbac` enabled in MCH (disabled by default)
- MCRA and ClusterPermission CRDs registered
- IDP users configured (HTPasswd, LDAP) via OAuth cluster config

## Hub-Side Components

### MCRA Operator

- **Namespace:** MCH namespace
- **Manages:** MultiClusterRoleAssignment CRDs

Reconciliation flow:
1. Watches MCRA resources
2. Evaluates scope (global, cluster set, cluster, project)
3. Generates ClusterPermission in each target cluster's namespace on hub
4. ClusterPermission triggers ManifestWork generation

Internal state management:
- Caches MCRA resources in memory for optimistic updates
- Concurrent PATCH handling is a known weakness (ACM-24737)
- Throttled resync to avoid hot-looping (ACM-25572 fix)

### console-api RBAC Endpoints

console-api serves RBAC data to the UI:
- `/rbac/identities` -- users, groups, service accounts
- `/rbac/roles` -- available roles and their definitions
- `/rbac/role-assignments` -- current MCRA assignments
- Depends on IDP (Identity Provider) for user enumeration

### acm-roles Addon

- **Addon name:** `acm-roles`
- **Default:** Deployed when fine-grained-rbac is enabled

Deploys predefined ACM ClusterRoles to managed clusters:
- `open-cluster-management:admin`
- `open-cluster-management:view`
- Virtualization-specific roles (kubevirt ClusterRoles)
- Custom roles defined via MCRA

### Search Integration

Search-api integrates with fine-grained RBAC:
- Queries filtered by user's MCRA-granted scope
- Aggregate API picks up kubevirt roles from ClusterPermission
- Fleet Virt tree view depends on RBAC-filtered search results
- Fine-grained users can search ~22 resource kinds with ~90 fields

## Spoke-Side Enforcement

### ManifestWork-Based Propagation

ClusterPermission on hub triggers ManifestWork creation:
1. ManifestWork contains Role, RoleBinding, ClusterRole, ClusterRoleBinding specs
2. work-agent on spoke applies the ManifestWork payload
3. Standard Kubernetes RBAC enforcement takes effect on spoke

### What Gets Created on Spoke

Per MCRA assignment, the spoke receives:
- **ClusterRoleBinding** for cluster-scoped access
- **RoleBinding** in each permitted namespace for project-scoped access
- **ClusterRole** (if custom role, via acm-roles addon)

## RBAC Flow (Console)

```
Admin creates MCRA via role assignment wizard
  -> wizard in roleAssignmentWizardHelper.ts
    -> constructs subject name, roles, target clusters
  -> ClusterPermission CR created on hub
  -> cluster-permission controller
    -> evaluates MCRA definition
    -> creates ClusterPermission resources on target spoke clusters
    -> spoke kubelets enforce the permissions
```

IDP user login: non-admin users authenticate via OAuth providers (HTPasswd, LDAP).
The console resolves the current user via `/api/username` endpoint.

SSE events: MCRA create/update/delete events are delivered to the UI via the SSE
endpoint in `events.ts`. If events for `MulticlusterRoleAssignment` kind are
dropped, the RBAC table doesn't auto-update.

## Configuration

### IDP Requirement

User enumeration (list users/groups for assignment) requires an Identity
Provider configured on the hub cluster:

```bash
oc get oauth cluster -o jsonpath='{.spec.identityProviders}'
```

Without IDP: User Management tab renders but user/group lists are empty.
Role assignments can still be created manually by typing usernames.

### MCH Component

```yaml
spec:
  overrides:
    components:
    - name: fine-grained-rbac
      enabled: true
```

## Console Integration

RBAC pages: role assignment wizard at `/multicloud/infrastructure/clusters/managed`
(RBAC tab), permissions page, `/multicloud/user-management/identities`,
`/multicloud/user-management/roles`.

---

## Known Failure Modes

- Subject name corruption: wizardHelper appends `-system` to the subject name
- SSE event dropping: events.ts filters out MCRA events silently
- Username reversal: username.ts reverses the name parts (kube:admin -> admin:kube)
- IDP authentication failure: non-admin users stuck on login form

## Cross-Subsystem Dependencies

| Dependency | Why |
|---|---|
| Console | RBAC UI is part of Access Management in console |
| console-api | Serves RBAC endpoints for UI |
| Search | RBAC-filtered resource views use search-api |
| Infrastructure (klusterlet) | ManifestWork delivery requires spoke connectivity |
| IDP (OAuth) | User/group enumeration for role assignments |
| addon-manager | Deploys acm-roles addon to spokes |

## What Depends on RBAC

| Consumer | Impact When RBAC Is Down |
|---|---|
| Fleet Virtualization UI | VM access control broken, users see wrong VMs |
| Console User Management | Tab absent or non-functional |
| Search filtered views | RBAC users may see too many or too few resources |
| Spoke Role/RoleBindings | Permissions not propagated, users can't access resources |
