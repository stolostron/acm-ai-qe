Diagnostic & Remediation Agent for ACM Hub Clusters

AI-powered health diagnostician built on Claude Code. The agent understands ACM architecture at the engineering level — operators, addon framework, dependency chains, known bugs with JIRA references — and uses that knowledge to diagnose root causes, not just report symptoms. Diagnosis is strictly read-only. When cluster-fixable issues are found, the agent presents a structured remediation plan and asks for explicit approval before making any changes.

Key capabilities: 6-phase diagnostic pipeline, 4 depth levels (quick pulse to deep audit), 54 knowledge files covering 12 ACM subsystems, 12-layer diagnostic model (vertical root cause tracing), 11 dependency chain traces, 13 diagnostic trap guards, cluster introspection (8 live metadata sources for reverse-engineering dependencies from any operator), version-aware known-issue matching, self-healing knowledge discovery, evidence-based diagnosis with confidence levels. Phase 1 checks operator deployment health (catches MCH operator at 0 replicas). Phase 3 uses layer-organized health checks — foundational layers (network, storage) are checked BEFORE component layers (operators, pods) to catch root causes before symptoms mislead. Phase 5 traces both horizontally (dependency chains) and vertically (12 infrastructure layers). Phase 6 uses layer-based fallback when no playbook matches.

Diagram 6: Dependency Chain Tracing

Phase 5 (Correlate) traces dependencies using three layers. Layer 1: 11 curated chains (8 core shown below + app-lifecycle, virtualization, submariner) cover ACM subsystems with hand-written impact analysis, tracing procedures, and JIRA references. Layer 2: Cluster introspection discovers chains dynamically from live metadata — works for any operator, not just ACM. Layer 3: The knowledge graph (neo4j-rhacm) provides 370 components and 541 dependency relationships. Real clusters have far more dependency paths than these 11 — the curated chains are the starting point, not the limit.
